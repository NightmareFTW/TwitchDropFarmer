# Twitch Drop Farmer v2.2.34

Esta versao reduz a frequencia com que o fallback invisivel de browser e mesmo necessario, na sequencia do trabalho sobre o integrity check nas versoes 2.2.x anteriores.

## Destaques

- Investigadas quatro formas de reduzir a dependencia do fallback QWebEngine; implementadas tres, uma delas descartada apos teste directo.
- Descartada: obter `twitch.tv/drops/campaigns` como HTML simples. Confirmado por pedido HTTP directo -- a pagina nao tem dados de campanhas embutidos; e uma SPA totalmente renderizada no cliente, sem injecao de dados no lado do servidor para fazer scraping.

## Adicionado

- **Perfil de browser persistente**: `_get_browser_profile` passa a criar uma unica `QWebEngineProfile` nomeada e persistida em disco (verificado `isOffTheRecord=False` contra um `QApplication` offscreen real), partilhada por todas as chamadas de fallback (listagem, detalhe por-campanha, scraper DOM, resgate de drops), em vez de um perfil anonimo novo a cada chamada. Cookies e local storage passam a persistir entre chamadas e reinicios da app, o que deve parecer uma sessao continua ao sistema anti-bot da Twitch, em vez de uma nova a cada ciclo de polling.
- **Captura e reaproveitamento do token Client-Integrity**: o script de intercecao de GQL ja existente passa tambem a capturar o header `Client-Integrity` que o proprio JavaScript da Twitch anexa aos seus pedidos `/gql` dentro do browser. Esse token passa a ser oferecido nos nossos pedidos HTTP directos durante ate ~4 minutos (limpo de imediato se um pedido que o use ainda assim for rejeitado), permitindo que varios ciclos de polling seguidos tenham hipotese de suceder sem invocar o browser.
- **Circuit breaker para DropCampaignDetails**: quando o `ViewerDropsDashboard` ja confirmou, no mesmo ciclo, que o integrity check esta bloqueado, e nao ha token utilizavel, o ciclo de repeticao do `DropCampaignDetails` (ate 2 identidades x 2 perfis de header x chunk -- ate ~8 pedidos e 16 linhas de log desperdicadas por ciclo) passa a ser saltado em vez de repetir uma falha ja confirmada.

## Resultado

A versao 2.2.34 deve reduzir, mas provavelmente nao eliminar, a frequencia com que "a tentar fallback do browser no thread principal" aparece -- a causa de fundo continua a ser a aplicacao intermitente do integrity check pela propria Twitch, que nenhuma correcao do lado do cliente consegue controlar por completo.
